 Creating and Defining Cinder Volume Types

In this section, we'll create a variety of Cinder Volume Types that leverage both the default capabilities of each driver, as well as the NetApp-specific extra specs described in Table 4.6, “NetApp-supported Extra Specs for use with Cinder Volume Types”.

  • The iscsi type will provision Cinder volumes onto any backend that uses the iSCSI storage protocol (in this example, that would be [cdot-iscsi] and [eseries-iscsi]).
  • The nfs type will provision Cinder volumes onto any backend that uses the NFS storage protocol (in this example, that would be [cdot-nfs]).
  • The gold type will provision Cinder volumes onto any backend that has a SnapMirror relationship (in this example, that would be [cdot-nfs], although only one of the four NFS exports defined in /etc/cinder/nfs_shares has this support).
  • The silver type will provision Cinder volumes onto any backend that has deduplication enabled (in this example, that would be [cdot-nfs], although only one of the four NFS exports defined in /etc/cinder/nfs_shares has this support).
  • The bronze type will provision Cinder volumes onto any backend that has compression enabled (in this example, that would be [cdot-nfs], although only one of the four NFS exports defined in /etc/cinder/nfs_shares has this support).
  • The analytics type will provision Cinder volumes onto the [eseries-iscsi] backend.

We'll now look at the local NFS mounts that are present on the node that is running cinder-volume and look for the volumes that were created on NFS backends. By mapping the mountpoints to the directories where the volume files exist, we are able to associate that the volumes were created in the appropriate FlexVol volume that had the NetApp-specific features enabled that matched the Cinder volume type definitions.
